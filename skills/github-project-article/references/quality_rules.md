# GitHub 项目文章 Quality Rules

## 文章风格

- 使用程序员和真实使用者视角，而不是项目档案视角。
- 写成公众号项目分享风，开头直接进入痛点、效果或使用场景。
- 多写这个项目能解决什么问题、用起来是什么效果、适合哪些例子。
- 段落要有叙述感，避免把 README 功能点改写成清单。
- 技术细节服务于理解，不堆术语、不做无来源夸张判断。

## 禁用表达

- 根据 README
- 资料显示
- 数据可能变化
- 发现一个 XX star 项目
- 本文将
- 综上所述
- 值得关注
- 具有较高参考价值

## 标题规则

- 标题优先场景导向，例如写清楚项目解决的具体问题。
- 标题可以效果导向，例如突出节省时间、降低门槛、提升体验。
- 不硬编 star 增长、爆火趋势、全网刷屏等无法证实的信息。
- 不把仓库名和 star 数作为唯一卖点。

## 配图规则

- README 图片优先，尤其是界面截图、架构图、使用效果图。
- 没有 README 图片时，用 GitHub README 页面截图兜底。
- 不强行生成装饰图，不用与项目功能无关的氛围图。
- 图片必须服务于理解项目效果，不能只是填充版面。

## 发布规则

- 发布文章只保留项目地址。
- 不堆参考链接、新闻链接、README 链接和无关来源。
- 发布包图片不能溢出，路径要可访问，Markdown 引用要能正常渲染。
- 文章结尾自然收束，不使用模板化总结口号。
